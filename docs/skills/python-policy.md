# Python Policy（全プロジェクト共通：安全自律ルール）

## Purpose（目的）
- Pythonは「検算・集計・変換・監査」に使い、プロダクション変更の事故を減らす
- AIが毎回迷わず、安全側に倒れる判断を自動で行う

---

## 1) Pythonを使う場面（自律トリガー：YES）
以下の条件に1つでも当てはまる場合、AIは **Pythonを優先** してよい：

- 金額/税/控除/時間/日数など「正しさ」が重要な計算
- Excel/CSVの整形、差分比較、集計、検算
- 大量データの変換（名前揺れ、SKU整形、ID付与など）
- ロジックが複雑で、サンプルデータで検証した方が早い
- バグ再現のための最小ケース生成（入力→出力の検証）

---

## 2) Pythonを使わない場面（自律トリガー：NO）
以下に当てはまる場合、AIは **Pythonを使わず** 通常実装に寄せる：

- UI微修正（CSS/表示/簡単なフォーム）
- Next.js / Supabase / Route Handlerの修正そのもの
- DBスキーマ変更（Pythonより migration / SQL を優先）
- 外部APIや秘密情報を含むデータを扱う必要がある（キー/個人情報）

---

## 3) 安全ガード（Python実行前チェック）
AIはPythonを使う前に必ず以下を確認する：

- .env* / APIキー / トークン / 個人情報（住所・電話・氏名）が入力データに含まれていない
- 実ファイルを書き換える処理は避け、まず「読み取り→出力」を優先
- 破壊操作（削除・上書き）は禁止（必要ならバックアップ手順を先に提示）
- 出力は再現可能（入力・前提・計算式）で説明できること

---

## 4) 出力フォーマット（Pythonを使った場合の回答形式）
Python検証を伴う回答は必ずこの順で出す：

1. Input assumptions（入力と前提）
2. Result（計算結果 / 検算結果）
3. Implementation rule（実装へ落とすルール：純関数/境界条件）
4. Edge cases（端数・欠損・例外・丸め）

---

## 5) プロジェクト別の位置づけ（使い方）
- E-STEP-TOOL：給与/台帳/税 → Pythonは「検算」専用（事故防止）
- E-NEXUS / antigravity-app：SKU/出品データ/CSV → Pythonは「整形・監査」専用
- 共通：Pythonで出した結論は、最終的に **実装ルール（docs or codeコメント）** に固定する
